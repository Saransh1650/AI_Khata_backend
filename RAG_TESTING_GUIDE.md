# RAG Memory System Testing Guide\n\nThis guide shows you how to test the RAG memory system with sample data to understand how it works and document its behavior.\n\n## Quick Start\n\n### 1. Setup Database Tables\n`bash\npsql $DATABASE_URL -f src/config/rag_memory.sql\n`\n\n### 2. Run the Demo Script\n`bash\nnode src/test/ragDemo.js\n`\n\n### 3. Test APIs (after getting store ID and token)\n`bash\nchmod +x src/test/testRAGAPIs.sh\n./src/test/testRAGAPIs.sh\n`\n\n## What the Demo Shows\n\n### Sample Data Overview\nThe demo creates a realistic grocery store scenario with:\n- **25+ transactions** over 4 weeks\n- **Common product combinations**: Milk+Bread+Butter, Tea+Biscuits, Rice+Oil\n- **Festival patterns**: Holi preparation with Milk+Sugar+Ghee+Sweets\n- **Regular customers** with repeat purchase behaviors\n\n### Learning Progression\n\n#### Stage 1: Basic Learning (5-10 transactions)\n`sql\nSELECT * FROM shop_memory WHERE store_id = 'your-store-id';\n`\n**Expected**: Basic product behavior entries with low confidence (0.3-0.5)\n\n#### Stage 2: Relationship Discovery (15-25 transactions)\n`sql\nSELECT * FROM product_relationships WHERE store_id = 'your-store-id';\n`\n**Expected**: Product pairs like:\n- Milk ↔ Bread (frequently_together, strength: 0.6+)\n- Tea ↔ Biscuits (frequently_together, strength: 0.5+)\n- Rice ↔ Oil (complementary, strength: 0.4+)\n\n#### Stage 3: Experience Insights (25+ transactions)\n`sql\nSELECT * FROM experience_insights WHERE store_id = 'your-store-id';\n`\n**Expected**: High-level insights about shop identity and customer preferences\n\n## Database Table Structures\n\n### shop_memory Table\n| Column | Sample Value | Description |\n|--------|--------------|-------------|\n| memory_type | 'product_behavior' | Type of pattern learned |\n| context | 'Milk' | Product or context name |\n| memory_data | `{\"avg_quantity\": 2.3, \"performance_indicator\": \"high_volume\"}` | Learned behavior data |\n| confidence | 0.75 | How confident the system is (0-1) |\n| frequency | 8 | How many times this pattern was seen |\n\n### product_relationships Table\n| Column | Sample Value | Description |\n|--------|--------------|-------------|\n| product_a | 'Milk' | First product |\n| product_b | 'Bread' | Related product |\n| relationship_type | 'frequently_together' | Type of relationship |\n| strength | 0.78 | Relationship strength (0-1) |\n| occurrences | 12 | How many times seen together |\n\n### experience_insights Table\n| Column | Sample Value | Description |\n|--------|--------------|-------------|\n| insight_category | 'shop_identity' | Category of insight |\n| title | 'Core Product Strengths' | Human-readable title |\n| description | 'This shop consistently performs well with: Milk, Bread, Tea' | Detailed description |\n| confidence | 0.85 | Confidence in this insight |\n\n## API Response \"Butter\",\n      \"reason\": \"Customers buying Milk often also want Butter\",\n      \"suggestion\": \"Consider stocking Butter to increase basket size\"\n    }\n  ]\n}\n`\n\n### Sales Expansion the Learning Process\n\n### Automatic Learning Triggers\n1. **Every New Transaction**: Basic product behavior learning\n2. **10 Transactions**: First relationship discovery\n3. **25 Transactions**: Deep relationship analysis\n4. **50 Transactions**: Experience insights generation\n5. **Every 50 after 100**: Ongoing relationship updates\n\n### Memory Confidence Evolution\n- **0.20-0.40**: Initial patterns (low confidence)\n- **0.40-0.60**: Emerging patterns (moderate confidence)\n- **0.60-0.80**: Established patterns (high confidence)\n- **0.80-1.00**: Proven patterns (very high confidence)\n\n### Relationship Types Discovered\n1. **frequently_together**: Products bought in same transaction\n2. **sequential**: Product B often bought after Product A\n3. **complementary**: Products that amplify each other's sales\n4. **seasonal_pair**: Products bought together during festivals\n\n## Testing Different Scenarios\n\n### Scenario 1: New Store (0-10 transactions)\n**Expected Behavior**: \n- Traditional AI guidance used\n- Basic memory entries with low confidence\n- No product relationships yet\n- Guidance focuses on inventory status\n\n### Scenario 2: Learning Store (10-25 transactions)\n**Expected Behavior**:\n- Mixed traditional + RAG guidance\n- Some product relationships discovered\n- Moderate confidence patterns\n- Beginning to show cross-sell opportunities\n\n### Scenario 3: Experienced Store (25+ transactions)\n**Expected Behavior**:\n- Full RAG-driven guidance\n- Strong product relationships\n- Experience insights available\n- Festival guidance based on shop history\n- Proactive revenue recommendations\n\n## Performance Metrics\n\n### Memory Health Levels\n- **Weak**: < 5 memory entries\n- **Developing**: 5-10 entries, some relationships\n- **Good**: 10-20 entries, 5+ relationships\n- **Excellent**: 20+ entries, 10+ relationships, insights\n\n### API Response Times\n- Memory recommendations: < 200ms\n- Sales expansion: < 300ms\n- Experience guidance: < 500ms\n- Health check: < 100ms\n\n## Common Patterns You'll See\n\n### Product Relationships\n1. **Milk + Bread + Butter** (breakfast combo)\n2. **Tea + Biscuits + Sugar** (tea time)\n3. **Rice + Oil + Onions** (cooking essentials)\n4. **Festival combinations** (context-specific)\n\n### Shop Identity Insights\n1. \"Core Product Strengths\" - top performing products\n2. \"Strong Product Pairings\" - consistent combinations\n3. \"Customer Preferences\" - behavioral patterns\n\n### Business Intelligence\n1. Peak day identification\n2. Product performance trends\n3. Seasonal adaptation patterns\n4. Revenue optimization opportunities\n\n## Debugging Tips\n\n### If Memory Isn't Learning\n- Check transaction line items are properly inserted\n- Verify products have consistent naming\n- Ensure multiple products per transaction\n- Check confidence thresholds in queries\n\n### If Relationships Are Weak\n- Need more transaction variety\n- Increase product combination diversity\n- Allow more time for pattern emergence\n- Check relationship discovery parameters\n\n### If APIs Return Errors\n- Verify database schema is created\n- Check store ownership permissions\n- Ensure sufficient transaction data\n- Review server logs for details\n\n## Documentation Tips\n\n1. **Screenshot Database Tables**: Show before/after memory states\n2. **Capture API Responses**: Document different confidence levels\n3. **Show Learning Progression**: Demonstrate how recommendations improve\n4. **Compare Traditional vs RAG**: Side-by-side guidance examples\n5. **Festival Season Demo**: Show memory-driven seasonal recommendations\n\n---\n\n*This testing guide helps you understand and document the complete RAG memory system behavior with real data examples.*
